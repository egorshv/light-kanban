# План технической реализации: light-kanban (MVP)

> Версия 1.0 · Основание: [docs.md](docs.md) v0.1 · Статус: утверждён к реализации

Документ фиксирует стек, архитектурные решения (продолжая нумерацию ADR из docs.md §7) и разбивку работ на фазы. Требования, доменная модель, схема БД и API-контракт **не дублируются** — они остаются в docs.md; здесь только то, чего там нет: *как именно строим*.

---

## 1. Зафиксированные решения по открытым вопросам

Ответы на вопросы из приложения docs.md, согласованы с владельцем продукта:

| № | Вопрос | Решение |
|---|--------|---------|
| 1 | Автосохранение карточки vs кнопка «Сохранить» | **Автосохранение**: `PATCH /tasks/{id}` с debounce ~500 мс на изменение поля / blur. Соответствует принципу «скорость важнее функций». |
| 2 | Архивация досок в MVP | **Да, в MVP**: `PATCH {archived: true}` → `archived_at`, список скрывает архивные без `?include_archived=true`. Поле уже в схеме, стоимость низкая. |
| 3 | «Финальная» колонка с автопростановкой `completed_at` | **Да, в MVP**: булев флаг `is_final` у колонки (единственное расширение схемы §5, см. ADR-006). Перенос задачи в финальную колонку проставляет `completed_at`, перенос из неё — сбрасывает. |
| — | Фильтр/поиск задач (US-C5, Should) | **Да, в MVP**: `GET /boards/{id}/tasks?q=&priority=&due_before=` + строка поиска в шапке доски. |
| 4 | Кросс-досочные связи в UI | **API — да, UI — только текущая доска**: автокомплит в форме добавления связи ищет по задачам текущей доски; связи между досками остаются доступны через API/MCP. Модель не ограничивается. |

Остальные Should/Could (JSON-экспорт, тёмная тема, `Idempotency-Key`, адаптив под планшет) — **пост-MVP** (см. §5).

## 2. Архитектурные решения

ADR-001 (вся доменная логика в сервисном слое), ADR-002 (SQLite через репозиторий), ADR-003 (без WebSocket, оптимистичный UI) — приняты в docs.md §7 и не пересматриваются. Ниже — новые решения.

### ADR-004. Стек бэкенда: Python + FastAPI

- Python 3.12+, FastAPI, Pydantic v2, uvicorn.
- Инструменты: **uv** (зависимости и venv), **ruff** (линт + формат), **pytest** (тесты).
- Обоснование: декларативная валидация и OpenAPI из коробки закрывают значительную часть требований §6 (коды 400/422, схемы запросов) почти бесплатно; вариант A из docs.md §7 прямо называет FastAPI первым кандидатом.

### ADR-005. Доступ к БД: stdlib `sqlite3`, без ORM

- Репозиторный слой пишет явный SQL по фиксированной схеме из 4 таблиц (docs.md §5). ORM на этом объёме — лишняя зависимость и лишний слой косвенности.
- Настройки соединения: `PRAGMA journal_mode=WAL`, `PRAGMA foreign_keys=ON`, `PRAGMA busy_timeout`.
- Миграции: файл `schema.sql` + версионирование через `PRAGMA user_version`; применяются автоматически при старте приложения. Отдельный инструмент миграций не нужен — один пользователь, одна БД.
- Один поток записи (uvicorn один воркер) — достаточно для одного пользователя, снимает вопросы конкурентной записи в SQLite.

### ADR-006. Слои и раскладка кода (реализация ADR-001)

```
api/       тонкие FastAPI-роутеры: Pydantic-схемы запросов/ответов,
           вызов сервиса, маппинг доменных исключений → HTTP
services/  ВСЯ доменная логика: проверка циклов (DFS по типу связи),
           пересчёт позиций в транзакции, WIP-предупреждения, каскады,
           «последняя колонка» (409), is_final → completed_at, валидация
repo/      SQL и только SQL; никаких правил
```

- Доменные ошибки — иерархия исключений (`NotFound`, `Conflict`, `DomainRuleViolation(code, message)`); единый FastAPI exception handler маппит их на 404/409/422 в формате `{"error": {"code", "message"}}` из §6. Роутеры сами кодов не назначают.
- Сервисы принимают соединение/транзакцию и **не знают про HTTP** — MCP-сервер этапа 2 вызывает их in-process без изменений.
- Расширение схемы §5 (единственное): `columns.is_final INTEGER NOT NULL DEFAULT 0` — под решение №3 из раздела 1.

### ADR-007. Фронтенд: React + Vite + TypeScript

- Данные и мутации — **TanStack Query**: оптимистичные обновления с откатом и `invalidate`/refetch доски при ошибке — ровно паттерн ADR-003, из коробки. Отдельный state-менеджер не нужен.
- Drag-and-drop — **@dnd-kit**: UX-спека §8 требует явную анимацию «куда встанет», нативный HTML5 DnD этого не даёт.
- Других runtime-зависимостей не добавляем без отдельного решения; markdown в описании задачи — одна лёгкая библиотека рендера (например, `marked`) с санитизацией.

### ADR-008. Локализация: словарь-модуль, без i18n-фреймворка

Все строки UI — в одном модуле `frontend/src/i18n/ru.ts` (плоский типизированный словарь). Требование §4 «тексты вынесены в словарь для будущей i18n» выполняется; фреймворк (i18next и т.п.) добавим, только когда появится второй язык.

### ADR-009. Раздача, конфигурация, деплой

- FastAPI раздаёт собранную SPA через `StaticFiles` (SPA-fallback на `index.html`) — один процесс, один порт, как рекомендует §7.
- Один **multi-stage Dockerfile**: стадия node (сборка фронта) → стадия python (uv sync + копия статики). `docker-compose.yml` — один сервис, volume для файла SQLite.
- Конфигурация через env: `KANBAN_DB_PATH`, `KANBAN_TOKEN` (пусто ⇒ аутентификация выключена — локальный режим), `KANBAN_LOG_LEVEL`. Проверка bearer-токена — одна FastAPI-dependency на роутер.
- HTTPS — забота reverse-proxy пользователя (§4), в приложении не реализуется.

### ADR-010. Наблюдаемость и события домена

- stdlib `logging` с JSON-форматтером (структурные логи §4), уровень из `KANBAN_LOG_LEVEL`; access-логи запросов через middleware.
- Сервисный слой с первого дня логирует доменные события (`task.created`, `task.moved`, `task.updated`, …) отдельным логгером — это задел под календарную синхронизацию этапа 3 (docs.md §9), где потребуется журнал изменений.

### Технические фиксации (повтор из docs.md для полноты)

- ID — uuid4 как TEXT; временные метки — ISO 8601 UTC-строки, генерируются в сервисном слое.
- Позиции — целые числа, полный пересчёт при перестановке, строго внутри транзакции.
- `tasks.column_id` — без `ON DELETE CASCADE`; удаление колонки — два явных сценария API (`?move_tasks_to=` либо 409).
- WIP-лимит мягкий: ответ `move`/`create` несёт поле-предупреждение, операция не блокируется.

## 3. Структура репозитория

```
light-kanban/
├── backend/
│   ├── app/
│   │   ├── api/          # роутеры: boards, columns, tasks, links + errors.py (handler)

│   │   ├── repo/         # SQL-слой + db.py (соединение, миграции)

│   ├── tests/
```

## 4. Фазы реализации

Фазы последовательны; каждая заканчивается проверяемым критерием готовности. Внутри фазы 3 возможна работа параллельно с хвостом фазы 2 (контракт API уже зафиксирован в §6).

### Фаза 0 — Скаффолдинг и «пустой прогон»

**Состав:** uv-проект бэкенда; Vite-проект фронтенда; `schema.sql` (схема §5 + `is_final`) и применение миграций при старте; `GET /api/v1/health`; Dockerfile + docker-compose; Makefile (`dev`, `test`, `lint`, `build`); каркас CI (прогон тестов на чистой БД — требование §10).

**Критерий готовности:** `docker compose up` поднимает контейнер, отдаёт пустую SPA и `health`; файл БД создаётся со схемой.

### Фаза 1 — Ядро домена (repo + services + unit-тесты)

**Состав:** репозитории и сервисы для всех четырёх сущностей. Все доменные правила:
- пересчёт позиций колонок и задач (перемещение, вставка, удаление) в транзакции;
- проверка циклов DFS для `blocks`/`subtask_of`; максимум один родитель для `subtask_of`; запрет self-link и дубликатов;
- каскады: доска → колонки/задачи/связи; задача → связи;
- удаление колонки: перенос задач в целевую / отказ при наличии задач; запрет удаления последней колонки;
- WIP: вычисление предупреждения; `is_final` → простановка/сброс `completed_at`;
- архивация досок; валидация длин и enum-полей.

Unit-тесты на in-memory SQLite покрывают каждое правило — это «ядро внимания» пирамиды §10.

**Критерий готовности:** все доменные правила из docs.md (§2, §3, §5) выражены в сервисах и покрыты зелёными unit-тестами; в `api/` — ни одного правила.

### Фаза 2 — REST API + contract-тесты

**Состав:** все эндпоинты §6 поверх сервисов; единый формат ошибок через exception handler; bearer-dependency; фильтры `q`/`priority`/`due_before`; `GET /boards/{id}` одной выборкой (колонки + задачи, требование < 500 мс на 100 задач). Contract-тесты: на каждый endpoint happy path + основные ошибки (400/401/404/409/422), прогон на чистой БД.

**Критерий готовности:** контракт §6 полностью реализован и зелёный в contract-тестах; ручная проверка OpenAPI (`/docs`).

### Фаза 3 — Фронтенд

**Состав** (порядок = порядок реализации):
1. Каркас: роутинг (список досок / доска), API-клиент, словарь `ru.ts`, обработка ошибок (toast + refetch доски).
2. Список досок: сетка карточек, создание (с флагом «пустая»/колонки по умолчанию), переименование, архивация, удаление с подтверждением «будут удалены N задач».
3. Доска: лента колонок, создание/редактирование колонки (имя, цвет, WIP, is_final), счётчик `n/WIP`, однострочная форма быстрого создания задачи (Enter/Esc), удаление колонки с выбором «перенести/удалить».
4. DnD задач и колонок (@dnd-kit) с оптимистичным обновлением и откатом; предупреждение при переносе в колонку с достигнутым WIP.
5. Карточка задачи — боковая панель: inline-заголовок, markdown-описание, приоритет, срок; автосохранение с debounce; секция «Связи» по типам с обратными формулировками, автокомплит по текущей доске, удаление связи.
6. Поиск/фильтр в шапке; индикаторы на карточке (полоса приоритета, просроченный срок, 🔒 «заблокирована» при активных блокерах); хоткеи `N`, `/`, `Esc`; пустые состояния.

**Критерий готовности:** все Must-stories §3 (+ выбранные Should) проходятся вручную в браузере против реального API.

### Фаза 4 — Интеграция, e2e, Definition of Done

**Состав:** e2e Playwright — 4–5 сценариев §10 (доска → колонка → задача → перетащить → связать → удалить доску) против docker-compose-сборки; README с запуском ≤ 3 команды; ручная проверка бэкапа/восстановления копированием файла БД; прогон DoD-чеклиста §10; обновление `CLAUDE.md` актуальными командами (требование самого CLAUDE.md).

**Критерий готовности:** все 6 пунктов Definition of Done из docs.md §10 выполнены.

## 5. Что сознательно НЕ делаем в MVP

| Что | Почему | Когда |
|-----|--------|-------|
| WebSockets / live-обновления | ADR-003: один пользователь | не планируется |
| Fractional indexing позиций | §5: полный пересчёт достаточен при личных объёмах | при деградации на >5 000 задач |
| ORM / SQLAlchemy | ADR-005: 4 таблицы, явный SQL проще | при миграции на другую СУБД (не планируется) |
| i18n-фреймворк | ADR-008: словарь-модуль достаточен | при втором языке |
| `Idempotency-Key` на create | Could в §6 | пост-MVP по необходимости |
| JSON-экспорт/импорт | Should в §4; бэкап = копия файла БД уже работает | пост-MVP, кандидат на этап 2+ |
| Тёмная тема | Should в §8 | пост-MVP |
| Мобильный/планшетный адаптив | Could в §4 | пост-MVP |

## 6. Готовность к пост-MVP

- **Этап 2 (MCP):** благодаря ADR-006 MCP-сервер — механическая задача: каждый tool (`list_boards`, `get_board`, `create_task`, `move_task`, …) — тонкая обёртка над уже существующей функцией сервисного слоя, вызываемой in-process; доменные исключения уже несут человекочитаемые `code` + `message`, которые агент может пересказать пользователю. Bearer-токен переиспользуется.
- **Этап 3 (календарь):** `due_date` уже в модели; iCal-фид (`/calendar.ics`) — один новый роутер над существующим сервисом задач. Лог доменных событий (ADR-010) с первого дня даёт материал для будущей двусторонней синхронизации.
